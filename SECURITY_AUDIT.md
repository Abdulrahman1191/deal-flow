# Security Audit — Auth / Authorization / Data-Access Layer

**Remediation status (follow-up to this audit, same issue #25 thread):** F1–F8
have been fixed in code — see the corresponding files for details. F1: dev
bypass and API-docs exposure now fail closed by default (`ENV` must be
explicitly `dev`). F2: every `/assessments/*` endpoint — including the
`GET /send-queue` list — is now scoped to `Lead.owner_email == user.email`
(via `_get_card_and_lead()` for the id-addressed endpoints, and an inline
`Lead.owner_email` filter for `/send-queue`), with `test_assessment_cross_user_returns_404`
and `test_send_queue_excludes_other_users_items` regression tests. F3: the Copper
webhook now rejects (401) whenever the signature is missing/invalid or the
secret is unset, via the shared `verify_webhook_signature()`. F4: the SPA
fallback resolves and confines the candidate path to `frontend-dist` before
serving. F5: outbound website fetches are blocked from targeting
private/loopback/link-local/reserved IPs, including on redirect hops. F6: the
unused `python-jose`/`passlib`/`bcrypt` pins were removed. F7: `/health` no
longer echoes exception `repr()` to the caller. F8: a security-headers
middleware now sets `X-Content-Type-Options`, `X-Frame-Options`,
`Referrer-Policy`, `Strict-Transport-Security`, and a `Content-Security-Policy`
on every response. The findings below are left as originally written, as the
historical record of what was found.

**Scope:** `backend/app` authentication, authorization, per-user data isolation,
admin gating, Copper webhook ingestion, secrets handling, CORS/headers/docs
exposure, injection surfaces, and pinned dependency risk.

**Method:** Static review of the FastAPI backend as committed on this branch,
plus local, offline reproduction of two findings (F1's environment-variable
default, and F5's path-traversal) using throwaway scripts against a copy of
the affected code — **no requests were made against the live deployment**,
per the issue's "no penetration testing / no network activity" constraint.

**Deliverable constraint respected:** this document is report-only. No
application code, config, dependency, or behavior was changed to produce it.

---

## Executive summary

The app's core authorization model — trust the platform proxy's
`X-Auth-Email` header, auto-provision a user, scope everything by
`owner_email`, gate admin tabs on an allow-list — is sound where it's
actually applied, and the `/leads/*` router (the most-audited surface,
per `test_multiuser_access.py`) is a good example of it done correctly,
including handling the tricky "owner asking for a colleague's lead" case
strictly. However, three areas break that model in ways that matter for a
platform-proxy-fronted, `@raed.vc`-internal tool: (1) the local-dev
`?fake_email=` bypass **fails open** — if the `ENV` environment variable is
ever unset or misspelled on the platform's deploy form, every request can
impersonate any user via a query string, no credentials required; (2) the
`/assessments/*` router — which includes the endpoint that sends a real,
external email — has **no per-user ownership check at all**, so any
authenticated teammate who learns another user's `lead_id` can read, approve,
override, or *send* another user's lead's outbound emails; and (3) the
Copper webhook signature check is structured so that **omitting the
signature header skips verification entirely** rather than rejecting the
request. Separately, static analysis surfaced a fourth, previously-unflagged
issue: the SPA static-file fallback in `main.py` builds a filesystem path
directly from the URL without confining it to the frontend build directory —
reproduced locally, this allows reading files outside `frontend-dist`, and
if that fallback is ever reachable pre-proxy-normalization (the same
proxy-bypass threat model the issue asks about in item 2), it can read
`/proc/self/environ`, which would hand over every secret in `Settings`
(Copper API key, DB URL with credentials, SMTP password, the Google service
account JSON) in one request. Findings are ranked below by realistic
exploitability against this specific deployment, not generic checklist
severity.

## Findings

| ID | Title | Severity | File:Line | Exploit scenario | Recommended fix |
|----|-------|----------|-----------|-------------------|------------------|
| F1 | `?fake_email=` dev bypass fails open when `ENV` is unset/misconfigured | **Critical** | `backend/app/services/auth.py:34` (`_LOCAL_DEV = os.getenv("ENV", "dev").lower() != "prod"`) | `_LOCAL_DEV` defaults to `True` whenever `ENV` is absent, empty, or misspelled (e.g. `production` instead of `prod`) — nothing in this repo sets `ENV=prod`; it depends entirely on the platform's deploy-form config, which is outside version control and has no fail-secure default. If that var is ever missing on the live deployment, any request to any endpoint with `?fake_email=owner@raed.vc` (or any arbitrary string) bypasses the proxy/header trust entirely and impersonates that user with zero credentials — full read/write on their leads, and (combined with F2) send access to *every* user's leads regardless of whose email was faked. This is exactly the scenario the issue asks to be evaluated, and the code has no second signal (e.g., requiring the header to also be absent, or checking for a platform-only marker) to prevent it. | Fail secure instead of fail open: default `_LOCAL_DEV` to `False` (i.e. flip the check to an explicit allow-list like `os.getenv("ENV") == "dev"`), and/or gate the fallback on an additional explicit flag (e.g. `ALLOW_FAKE_EMAIL=1`) that would never plausibly be set in the platform's prod env. |
| F2 | `/assessments/*` router has no per-user ownership scoping — cross-user IDOR, including real outbound email send | **Critical** | `backend/app/routers/assessments.py` — `_get_card()` (lines 31–41) and every `Lead` lookup in the file (e.g. lines 115, 199, 217, 276, 318, 397, 423, 458) all query by `lead_id`/`AssessmentCard.lead_id` alone, with **no** `Lead.owner_email == user.email` filter anywhere in the file | Any authenticated `@raed.vc` teammate who obtains another user's `lead_id` (visible in Copper custom fields the app writes back, in event-log payloads, via a shared link, or via a departed/former-access colleague) can: `GET /assessments/{lead_id}` to read another user's lead summary/draft; `POST /assessments/{lead_id}/approve`, `/override`, `/rate`, `/regenerate-draft`, `/reassess` to mutate another user's pipeline; and — most seriously — `POST /assessments/{lead_id}/send`, which calls `email_sender.send_email()` and actually delivers an email to the founder's real inbox, with no check that the caller owns the lead. This directly contradicts the ownership model the `/leads/*` router enforces and `test_multiuser_access.py` locks in (compare `test_lead_by_id_cross_user_returns_404`) — that test suite has no equivalent case for `/assessments/*`, which is why this gap wasn't caught. `_finalize_sent`'s Copper writes (opportunity conversion, archiving) would also fire for a lead the caller doesn't own. | Add `.join(Lead).where(Lead.owner_email == user.email)` (or an equivalent ownership check against the lead before touching the card) to every endpoint in this router, mirroring the pattern already used in `leads.py`. Add a `test_assessment_cross_user_returns_404`-style test analogous to the existing lead one. |
| F3 | Copper webhook HMAC verification is skipped (not rejected) when the signature header is absent or the secret is unset | **High** | `backend/app/routers/leads.py:106-108` (`if settings.copper_webhook_secret and x_copper_signature: if not _verify_copper_hmac(...): raise 403`) | The check is structured as "verify only if both a secret is configured **and** a signature header was sent." If `COPPER_WEBHOOK_SECRET` is empty (its default, per `config.py:21`) or an attacker simply omits `X-Copper-Signature`, verification is skipped entirely and the payload is processed as trusted input — no 401/403, straight into `ingest_lead`. An attacker who can reach `/api/v1/leads/ingest` (it requires no user auth, by design, since Copper has no Slack identity) can inject fabricated leads, attribute them to any known user via a forged `assignee_id` (`_owner_for_assignee`, line 37-51), or send `delete`/`update` events to archive or corrupt real leads. `verify_webhook_signature()` in `auth.py` (used nowhere — `leads.py` has its own duplicate `_verify_copper_hmac`) is at least written correctly (constant-time via `hmac.compare_digest`, returns `False` — i.e. reject — when the secret or signature is missing), so the primitive itself is fine; it's the router's call-site logic that's wrong. No replay protection exists either (no timestamp/nonce check), though the echo-loop guard in `copper_echo_guard.py` incidentally limits (but does not intend to prevent) some replay patterns. | Change the guard to unconditionally require and verify the signature: reject with 401 whenever `x_copper_signature` is missing or `copper_webhook_secret` is unset (fail closed, not skip). Drop the duplicate `_verify_copper_hmac` in favor of the one already in `auth.py`. Consider adding a timestamp check for replay protection. |
| F4 | SPA static-file fallback allows path traversal outside `frontend-dist` | **High** | `backend/app/main.py:94-116` (`spa_fallback`, specifically `candidate = _FRONTEND_DIST / spa_path` at line 103 and `candidate.is_file()` at 104) | Reproduced locally (not against the live site): built the exact handler in isolation and drove it via a raw ASGI scope (bypassing HTTP-client-side URL normalization, to model what a raw request the platform proxy forwards unmodified would look like) with `path="/../secret.txt"`. The handler resolved `candidate` to a file *outside* the intended `frontend-dist` directory and served its contents with `FileResponse` — confirmed 200 with the out-of-directory file's real content. `Path.__truediv__` does not collapse or validate `..` segments, and there's no `resolve()`-then-`is_relative_to()` check before the `is_file()`/`FileResponse` calls (contrast with the `/assets` mount two lines above, which uses Starlette's `StaticFiles`, itself hardened against this). Exploitability hinges on whether anything upstream (the platform proxy, or Uvicorn's own request-line parsing) normalizes `..`/percent-encoded dot-segments before the ASGI app sees them — that's outside this app's control and unverified here (no live traffic was sent), so the app currently provides **no defense-in-depth** of its own. If reachable — including via the same "proxy bypass" threat model the issue asks about in item 2 (direct container access, misrouted ingress) — the impact is severe: reading `/proc/self/environ` from within the container would dump every secret in `Settings` (`COPPER_API_KEY`, the full `DATABASE_URL` including its password, `SMTP_PASSWORD`, `GOOGLE_SERVICE_ACCOUNT_JSON`, `COPPER_WEBHOOK_SECRET`) in a single unauthenticated request, since this route requires no auth by design (it's the SPA catch-all). | Resolve `candidate` and verify it's still inside `_FRONTEND_DIST.resolve()` before serving (e.g. `candidate.resolve().is_relative_to(_FRONTEND_DIST.resolve())`), or — simpler and more robust — mount the whole SPA fallback through `StaticFiles(..., html=True)` / `check_dir`-style helper instead of hand-rolling path joins, matching what's already done for `/assets`. |
| F5 | SSRF surface in LinkedIn-discovery website fetch | **Medium** | `backend/app/services/research.py:29-37` (`scrape_linkedin_from_website`, `httpx.get(url, ..., follow_redirects=True)`) | `POST /leads/{lead_id}/find-linkedin` (any authenticated user, for their own lead) triggers a server-side `GET` to `lead.website`, a field populated from Copper CRM data (which can originate from inbound web-form submissions, i.e. attacker-influenced) with no scheme allow-list, no private/link-local IP block, and `follow_redirects=True`. A lead whose `website` is set to `http://169.254.169.254/...` or an internal service URL would cause the backend to fetch it server-side. Impact is bounded (response isn't returned verbatim to the caller — only a regex-extracted `linkedin.com` URL is persisted — so this is a blind SSRF, not a direct data-exfiltration one, and it requires a user to first get a malicious `website` value onto a lead via Copper), which is why this is Medium rather than High for this specific internal-tool deployment. | Validate `website` resolves to a public IP before fetching (block RFC1918/link-local/loopback ranges), and/or route through an egress proxy that already does this. |
| F6 | Legacy JWT/password-hashing dependencies remain pinned though unused | **Low** (dependency hygiene) | `backend/requirements.txt` — `python-jose[cryptography]==3.3.0`, `passlib[bcrypt]==1.7.4`, `bcrypt==4.0.1` | `python-jose` 3.3.0 has a known algorithm-confusion/DoS advisory in its ECDSA key-handling path (fixed upstream in later releases); grepping `backend/app` confirms neither `jose` nor `passlib`/`bcrypt` is imported anywhere in the current auth flow (the platform-proxy header-trust model replaced the old JWT/password login entirely — see the "Legacy" note in `auth.py`'s module docstring). They're pure attack-surface with no functional benefit today. **Note only, per issue scope — no upgrade/removal performed here.** | File a follow-up to remove these three packages from `requirements.txt` (no code references them) and run `pip-audit`/`safety` over the full pinned set for a complete CVE sweep — this review flagged only the packages with a clearly unused, historically-vulnerable profile; it is not a substitute for automated dependency scanning. |
| F7 | Unauthenticated `/health` endpoint echoes truncated exception `repr()` | **Low** | `backend/app/main.py:66-72` | `GET /health` requires no auth (by design, for the platform proxy / uptime monitors) and on a DB failure returns `repr(exc)[:200]` in the JSON body. SQLAlchemy generally redacts the password in its own exception string rendering, so this is unlikely to leak the DB credential today, but it's still unauthenticated internal-error detail (hostnames, driver error text) going out to anyone who can reach the endpoint, and it's fragile to a future change in what wraps that exception. | Log the full exception server-side; return a generic `{"status": "degraded", "db": "error"}` in the response body. |
| F8 | No response security-hardening headers set (CSP, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, HSTS) | **Low** | `backend/app/main.py:41-47` — the only middleware registered is `CORSMiddleware`; grepping the whole app for `Strict-Transport-Security`, `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, and `Referrer-Policy` returns zero hits, so no response, on any route, ever carries them | Every response — API JSON and the served SPA — is missing standard hardening headers. For this platform-proxy-fronted, `@raed.vc`-internal deployment the blast radius is smaller than a public app's would be: the proxy terminates TLS (so HSTS is arguably the proxy's job, though this repo has no way to confirm the proxy actually sets it, and setting it here too costs nothing), and the tool isn't meant to be embedded or reached by the public internet. It's not risk-free, though: with no `X-Frame-Options`/`frame-ancestors`, nothing stops the SPA from being iframed for a clickjacking/UI-redress attack against an authenticated `@raed.vc` session if a victim can be lured to a framing page; with no `X-Content-Type-Options: nosniff`, browsers may MIME-sniff responses that embed user-influenced, Copper-sourced text (lead/company names, notes) rendered by the frontend; and with no `Content-Security-Policy`, there's no second layer of defense if a stored-XSS vector is ever introduced via Copper-sourced fields (none was found in this audit, but CSP is precisely the control that caps the blast radius when one is missed later). This doesn't rise to F1–F4's severity given the internal, proxy-fronted threat model, but it's an unaddressed gap, not a deliberate/justified omission — hence a Low finding here rather than "reviewed but found safe." | Add a small middleware (or confirm and rely on the platform proxy) that sets `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY` (or `frame-ancestors 'none'` via CSP), `Referrer-Policy: same-origin`, `Strict-Transport-Security: max-age=...` if the proxy doesn't already, and a `Content-Security-Policy` scoped to the SPA's actual script/style/connect sources. Report-only per this audit's scope — not implemented here. |

## Reviewed but found safe

- **Per-user lead scoping (`/leads/*`)** — every list/get/patch/delete/export/archive/events/pitch-deck endpoint in `backend/app/routers/leads.py` filters on `Lead.owner_email == user.email`, including the id-addressed `GET /leads/{lead_id}`. This is locked in by `test_multiuser_access.py::test_lead_by_id_cross_user_returns_404`, which also confirms the owner does *not* get a backdoor to a colleague's lead — the strictest reasonable interpretation of "per-user isolation." Good pattern; F2 is exactly this same protection missing one router over.
- **Portfolio Intelligence (`/portfolio/*`)** — every endpoint calls `_owner_or_403()` (admin allow-list gate) *and* additionally scopes queries by `PortfolioCompany.owner_email == user.email` (e.g. `portfolio.py:161, 220, 259, 286, 304, 340`). Double-gated; no issue found.
- **Feedback inbox (`/feedback`)** — `submit_feedback` correctly records the *caller's own* email and requires no special privilege (intentional — anyone can submit feedback); `list_feedback` and `resolve_feedback` are both gated by `is_owner()`. Confirmed by `test_feedback_inbox_blocked_for_colleague` / `test_feedback_resolve_blocked_for_colleague`.
- **Overrides / training-data inbox (`/overrides/*`)** — both endpoints gate on `is_owner()`; deliberately not scoped per-user since it's a cross-user training-data aggregation used by the eval harness, which is the intended design (stated in the module docstring) rather than an oversight.
- **Briefings (`/briefings/*`)** — all three read endpoints filter on `DailyBriefing.owner_email == user.email`; `POST /generate` queues generation for the caller's own email only.
- **Admin gating primitive (`is_owner()` / `admin_email_set()`)** — defaults to exactly `{owner_email}` when `ADMIN_EMAILS` is unset (`config.py:99-103`), so admin is never accidentally granted to all users by an empty/missing env var — this one fails *closed*, unlike F1.
- **`verify_webhook_signature()` primitive itself (`auth.py:105-116`)** — uses `hmac.compare_digest` (constant-time) and correctly returns `False`/reject when the secret or signature is missing. The primitive is fine; F3 is about the router not always calling it.
- **CORS** — `CORSMiddleware` in `main.py` only allows the explicit `ALLOWED_ORIGINS` env list (defaulting to `localhost` dev origins), never `*`; combined with `allow_credentials=True` this is the correct posture (browsers reject `*` + credentials anyway), and since frontend and API are same-origin in production per the module docstring, CORS isn't actually load-bearing for the deployed app — it only matters for local dev against Vite. (Response security headers are a separate area — see **F8**, since none are currently set.)
- **API docs / OpenAPI exposure** — `main.py:28-33` sets `docs_url=None, redoc_url=None` whenever `_is_prod` is true (same `ENV=="prod"` check as F1 — see F1's note about that variable's fail-open default applying here too, though the blast radius of exposed docs is far smaller than F1's auth bypass).
- **SQL injection** — no string-interpolated SQL anywhere in the codebase; every `text(...)` call (`portfolio.py:395,411`, `overrides.py:87,100`) uses bound `:param` placeholders, and all ORM queries use SQLAlchemy's parameterized `select()`/`.where()`. Confirmed via repo-wide grep for f-string/`.format()`-built queries — the only `.format()` usage found is in `claude_agent.py` building LLM prompt text, unrelated to SQL.
- **Pitch-deck viewer (`GET /leads/{lead_id}/pitch-deck`)** — redirects to a Google Drive URL built from an opaque `pitch_deck_drive_id` column value, never from user input; the actual PDF text-extraction code path (`pitch_deck.py`) runs offline against a local sync script's file listing, not against any HTTP-request-supplied path. No traversal surface here — see F4 for the *actual* traversal bug found elsewhere, in the SPA fallback route, not this endpoint.
- **Secrets in logs/errors/responses** — grepped every `print(`/logging call alongside `secret|password|token|key|signature`; found none that log a secret value (all instances log a *copper_id* or an exception `repr()` of a non-credential failure, e.g. `[echo_guard] register failed`, `[copper_writer] convert ... failed`). No `.env` file or hardcoded credential pattern (`sk-...`, AWS keys, PEM private keys) is committed to the repo.
- **Rating gate (`_require_rating`)** — applies uniformly to `/approve` and `/mark-sent`/`/send` for every user regardless of admin status, per `test_rating_gate_applies_to_every_user`.

## Notes on methodology limits

- No requests were sent to `https://deal-flow.apps.raed.vc` or any other live host — F1 and F4 were reasoned about/reproduced entirely offline, against copies of the relevant code in a throwaway script, per the issue's explicit prohibition on live network activity.
- Dependency risk (F6) is a targeted note on the clearest case (unused, historically-vulnerable, easy to drop), not an exhaustive CVE sweep — recommend `pip-audit` as a follow-up for full coverage of `requirements.txt`.
- This audit did not review the frontend (`frontend/`) beyond confirming it has no independent authorization logic to bypass — the backend is the sole enforcement point, consistent with the platform-proxy trust model.
